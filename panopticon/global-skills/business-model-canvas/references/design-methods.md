# 设计方法与评估框架

本文档详细描述6大设计方法论、战略评估框架、工具和模板。

---

## 6大设计方法论详解

### 1. 客户洞察 (Customer Insight)

#### 目标
挖掘未被满足的潜在需求，理解客户的真实痛点和动机。

#### 方法与工具

##### 1.1 用户访谈

**准备工作：**
- 明确访谈目标：想了解什么？
- 确定访谈对象：5-10个目标用户
- 设计访谈提纲：开放式问题为主

**访谈提纲示例：**
1. 请描述一下您当前如何解决[问题]？
2. 在这个过程中，您遇到的最大挑战是什么？
3. 如果可以改进，您最希望改善什么？
4. 您曾经尝试过哪些替代方案？为什么没有持续？
5. 如果有完美的解决方案，它应该是什么样的？

**访谈技巧：**
- 避免"是/否"问题，多问"为什么"、"如何"
- 关注故事细节，鼓励用户举例
- 观察非语言信号（表情、肢体语言）
- 不要引导用户，保持中立
- 录音（征得同意）以便后续分析

**输出：** 访谈记录、需求清单

##### 1.2 问卷调查

**适用场景：**
- 需要批量收集用户反馈（50+用户）
- 需要量化数据验证假设
- 需要统计分析用户偏好

**问卷设计原则：**
- 问题数量：10-20题为宜，避免过长
- 题型搭配：单选、多选、开放题结合
- 逻辑顺序：从易到难，从一般到具体
- 避免引导性问题

**问卷结构示例：**
1. **基本信息**（2-3题）：年龄、职业、使用频率
2. **现状评估**（5-8题）：使用体验、满意度、痛点
3. **需求探索**（3-5题）：未被满足的需求、期望
4. **开放反馈**（1-2题）：建议、补充

**输出：** 问卷报告、统计分析结果

##### 1.3 观察研究

**方法：**
- **参与式观察：** 真实使用场景下观察用户行为
- **非参与式观察：** 站在用户外部观察
- **远程观察：** 通过屏幕共享、视频会议观察

**观察要点：**
- 用户如何使用产品/服务？
- 遇到哪些困难或困惑？
- 有哪些"变通"做法？
- 哪些功能被忽略？

**输出：** 观察记录、行为模式分析

##### 1.4 数据分析

**数据来源：**
- 用户行为数据（点击流、使用路径）
- 业务数据（转化率、留存率、复购率）
- 客服数据（工单类型、常见问题）
- 社交数据（评论、反馈、讨论）

**分析方法：**
- **漏斗分析：** 识别转化瓶颈
- **留存分析：** 识别用户流失原因
- **路径分析：** 识别用户行为模式
- **聚类分析：** 识别用户分群

**输出：** 数据分析报告、用户行为洞察

### 2. 创意构思 (Ideation)

#### 目标
生成多样化的创意方案，突破思维定势。

#### 方法与工具

##### 2.1 头脑风暴

**基本原则：**
- 延迟评判：不对任何创意进行批评
- 数量优先：先追求数量，再追求质量
- 鼓励荒诞：看似荒诞的创意可能隐藏价值
- 借力发挥：在他人的创意基础上发展

**操作步骤：**
1. 设定明确的目标和主题
2. 给定时间（如30分钟）
3. 快速记录所有创意（不少于10个）
4. 初步筛选（保留3-5个有潜力的）

**输出：** 创意清单（10+个）

##### 2.2 SCAMPER法

**七个维度：**

| 维度 | 思考问题 | 示例 |
|------|----------|------|
| **S**ubstitute（替代） | 什么可以替代？用什么替代？ | 用AI客服替代人工客服 |
| **C**ombine（合并） | 什么可以合并？ | 订阅+会员合并为VIP体系 |
| **A**dapt（调整） | 什么可以调整？ | 调整定价模式为动态定价 |
| **M**odify（修改） | 什么可以修改？ | 修改产品功能模块 |
| **P**ut to other use（其他用途） | 还有其他用途吗？ | 将技术开放给第三方使用 |
| **E**liminate（消除） | 什么可以消除？ | 消除免费层，转为纯订阅 |
| **R**earrange（重排） | 什么可以重排？ | 重排用户体验流程 |

**操作步骤：**
1. 逐一应用7个维度
2. 每个维度生成2-3个创意
3. 整合所有创意
4. 评估可行性

**输出：** SCAMPER创意清单

##### 2.3 类比法

**操作步骤：**
1. 选择类比对象（其他行业、其他领域）
2. 分析类比对象的商业模式
3. 提取可借鉴的要素
4. 应用到当前场景

**类比示例：**
- **订阅制类比：** 参考Netflix的订阅模式，应用于健身行业（ClassPass）
- **平台类比：** 参考Uber的平台模式，应用于货运领域（满帮）
- **长尾类比：** 参考Amazon的长尾模式，应用于教育领域（Coursera）

**输出：** 类比创意清单

##### 2.4 逆向思维

**操作步骤：**
1. 列出常规做法
2. 反向思考：如果完全相反会怎样？
3. 探索反向做法的可能性
4. 评估可行性

**逆向思维示例：**
- 常规：免费基础服务+付费增值 → 逆向：付费基础服务+免费增值
- 常规：集中式平台 → 逆向：去中心化网络
- 常规：由企业设计 → 逆向：由用户共同创造

**输出：** 逆向思维创意清单

#### 输出模板：创意清单

| 创意编号 | 创意描述 | 生成方法 | 可行性评估（1-5） | 优先级 |
|----------|----------|----------|-------------------|--------|
| C01 | [创意1描述] | [头脑风暴/SCAMPER/类比/逆向] | [1-5] | [高/中/低] |
| C02 | [创意2描述] | [头脑风暴/SCAMPER/类比/逆向] | [1-5] | [高/中/低] |
| ... | ... | ... | ... | ... |

### 3. 可视思考 (Visual Thinking)

#### 目标
将抽象的创意转化为可视化的方案，便于沟通与迭代。

#### 方法与工具

##### 3.1 商业模式画布草图

**画布布局：**

```
┌─────────────────────┬─────────────────────┬─────────────────────┐
│    客户关系         │     价值主张       │      客户细分       │
│                     │                     │                     │
└─────────────────────┴─────────────────────┴─────────────────────┘
┌─────────────────────┬─────────────────────┬─────────────────────┐
│    渠道通路         │                     │                     │
│                     │                     │                     │
├─────────────────────┼─────────────────────┼─────────────────────┤
│                     │                     │                     │
│   重要合作          │                     │     收入来源        │
├─────────────────────┼─────────────────────┤                     │
│   关键业务          │   核心资源          │                     │
│                     │                     │                     │
├─────────────────────┴─────────────────────┤                     │
│      成本结构                             │                     │
│                                             └─────────────────────┘
└───────────────────────────────────────────┘
```

**绘制技巧：**
- 先从核心开始（价值主张），再填充左右两侧
- 使用简洁的关键词，避免长句
- 用颜色或标记区分核心模块与次要模块
- 留出空白，便于后续迭代

##### 3.2 故事板 (Storyboard)

**目的：** 用场景化方式呈现用户旅程

**故事板结构：**
- 4-8个关键场景
- 每个场景包含：时间、人物、地点、事件
- 从用户视角讲述故事

**故事板示例：**
```
场景1：用户发现产品
时间：周一上午
人物：张三（白领）
事件：在朋友圈看到朋友分享的产品，点击了解

场景2：用户尝试产品
时间：周一上午
人物：张三
事件：注册账号，体验核心功能

场景3：用户遇到问题
时间：周一下午
人物：张三
事件：在某个功能遇到困难，寻求帮助

场景4：用户获得价值
时间：周二上午
人物：张三
事件：成功解决问题，获得价值

场景5：用户推荐
时间：周二晚上
人物：张三
事件：分享到朋友圈，推荐给朋友
```

**输出：** 用户故事板

##### 3.3 流程图 (Flowchart)

**目的：** 绘制关键业务流程

**常用符号：**
- 矩形：活动/步骤
- 菱形：决策点
- 圆角矩形：开始/结束
- 平行四边形：输入/输出

**流程图示例（用户注册流程）：**
```
[开始] → [打开APP] → [点击注册] → [输入手机号] → [发送验证码]
                                                   ↓
                                           [是否收到？]
                                              /      \
                                          [是]      [否]
                                           ↓          ↓
                                  [输入验证码]   [重新发送]
                                           ↓
                                  [设置密码] → [注册成功] → [结束]
```

**输出：** 业务流程图

##### 3.4 原型图 (Prototype)

**目的：** 绘制产品/服务的初步原型

**原型类型：**
- **纸质原型：** 手绘草图，快速验证
- **低保真原型：** 使用工具（Axure、Figma）绘制静态页面
- **高保真原型：** 接近真实产品的交互原型

**输出：** 原型图

### 4. 原型制作 (Prototyping)

#### 目标
构建最小可行性商业模型（MBM），快速验证核心假设。

#### MBM构建原则

**核心假设：**
1. 价值假设：用户是否认可这个价值？
2. 需求假设：用户是否有这个需求？
3. 盈利假设：是否能够盈利？

**最小化条件：**
- 最少资源：用最少的资源构建
- 最少功能：仅保留核心价值
- 最少渠道：用最少的渠道触达用户
- 最少客户：验证最小市场规模

#### 原型类型

##### 4.1 纸质原型

**适用场景：** 初期创意验证，快速迭代

**方法：**
- 手绘产品界面或服务流程
- 纸上模拟用户使用过程
- 快速修改和调整

**优点：** 快速、低成本、易于修改
**缺点：** 体验有限，难以模拟复杂交互

##### 4.2 数字原型

**适用场景：** 产品界面和交互验证

**工具：**
- Figma、Sketch、Axure（界面设计）
- Webflow（网站搭建）
- Bubble（无代码开发）

**方法：**
1. 绘制低保真原型（线框图）
2. 增加交互逻辑
3. 进行可用性测试
4. 迭代优化

##### 4.3 MVP（最小可行产品）

**适用场景：** 产品可行性验证

**MVP vs MBM：**
- **MVP：** 最小可行产品，侧重产品功能
- **MBM：** 最小可行商业模型，侧重商业逻辑

**MVP特征：**
- 功能最小化：仅包含核心功能
- 快速上线：2-8周完成
- 收集反馈：真实用户反馈

##### 4.4 商业原型

**适用场景：** 商业模式可行性验证

**类型：**
- **预售：** 在产品开发前，通过预售验证需求
- **众筹：** 通过众筹平台验证需求并获得资金
- **测试广告：** 投放少量广告，测试点击率、转化率
- **Landing Page：** 创建单页网站，测试用户兴趣

**核心假设：**
1. **价值假设：** [描述] - [如何验证]

3. **盈利假设：** [描述] - [如何验证]

### 5. 故事讲述 (Storytelling)

#### 目标
将商业模型转化为易于传播的商业故事，用于内部沟通、融资、市场推广。

#### 故事结构

##### 5.1 经典叙事结构

**三幕式结构：**
1. **第一幕：设定（现状）**
   - 描述当前的困境或机会
   - 引入主角（目标客户）
   - 识别问题

2. **第二幕：冲突（挑战）**
   - 描述尝试的解决方案
   - 说明为何失败或不够好
   - 提出新的可能

3. **第三幕：结局（价值）**
   - 介绍你的解决方案
   - 展示带来的价值
   - 展望未来

##### 5.2 商业故事框架

**框架：**
1. **问题：** 市场存在什么问题？
2. **用户：** 谁有这个问题？
3. **现有方案：** 他们如何解决？为什么不够好？
4. **我们的方案：** 我们如何解决？
5. **价值：** 带来什么价值？
6. **证据：** 有什么证据支持？

#### 故事讲述技巧

##### 5.3 用户视角

**原则：**
- 从用户视角出发，而非产品视角
- 描述用户场景，而非产品功能
- 关注用户获得的收益，而非产品特性

**示例：**
- ❌ 产品视角："我们的产品有AI推荐功能"
- ✅ 用户视角："用户不再需要花费数小时寻找感兴趣的内容，AI会自动推荐"

##### 5.4 场景化

**方法：**
- 用具体的场景描述代替抽象的概念
- 加入时间、地点、人物、事件
- 让读者能够"身临其境"

**示例：**
- ❌ 抽象："我们的产品提高了工作效率"
- ✅ 场景化："周一上午，张三打开电脑，不再需要像以前那样切换5个软件，现在所有工作都在一个界面上完成，节省了2小时"

##### 5.5 视觉化

**方法：**
- 配合图表、图片增强故事性
- 使用对比图（Before/After）
- 使用数据可视化

### 6. 情景推测 (Scenario Planning)

#### 目标
预设外部环境变化，制定应对策略，提高商业模式的韧性。

#### 情景设定

##### 6.1 情景类型

| 情景 | 描述 | 假设条件 |
|------|------|----------|
| 乐观情景 | 最有利的外部环境 | 市场快速增长、技术顺利发展、政策支持、竞争压力小 |
| 基准情景 | 正常发展的外部环境 | 市场稳步增长、技术正常发展、政策中性、竞争正常 |
| 悲观情景 | 最不利的外部环境 | 市场萎缩、技术受阻、政策限制、竞争激烈 |

##### 6.2 情景要素

**需要考虑的外部要素：**
- **市场：** 市场规模、增长速度、用户行为变化
- **技术：** 技术发展趋势、技术突破、技术壁垒
- **政策：** 法规变化、政府政策、行业标准
- **竞争：** 竞争者数量、竞争者策略、价格战
- **经济：** 宏观经济环境、利率、汇率
- **社会：** 社会趋势、人口结构、文化变化

#### 情景分析框架

##### 6.3 分析步骤

**步骤1：定义情景**
- 设定乐观、基准、悲观三种情景
- 为每个情景设定具体假设条件
- 明确情景的时间范围（1年、3年、5年）

**步骤2：分析影响**
- 分析每个情景对商业模式9大模块的影响
- 识别关键风险点
- 评估商业模式的韧性

**步骤3：制定应对策略**
- 针对每个风险点，制定应对策略
- 识别提前需要准备的条件
- 制定触发条件（何时启动应对策略）

**步骤4：监控与调整**
- 设定监控指标（KPI）
- 定期回顾（季度/年度）
- 根据实际情况调整情景

#### 2. 关键指标计算

**毛利率：**
```
毛利率 = （总收入 - 直接成本）/ 总收入 × 100%
```

**获客成本（CAC）：**
```
CAC = （营销费用 + 销售费用）/ 新增客户数
```

**客户终身价值（LTV）：**
```
LTV = 平均月收入 × 平均客户生命周期（月）
```

**LTV/CAC比率：**
```
LTV/CAC = LTV / CAC
```

**回本周期：**
```
回本周期 = CAC / （月度客户收入 - 月度客户服务成本）
```
